## 4. Known Limitations and Notes

### 4.1 color-ink-secondary Contrast on Dark Backgrounds

`color-ink-secondary` (`#5A4E47`) has a contrast ratio of approximately **1.6:1** against `color-ink` (`#231E1A`). This fails WCAG AA by a wide margin. It is present in the token set because it reads correctly on light backgrounds (`color-surface` context) — not on dark backgrounds.

**Rule:** Do not use `color-ink-secondary` for any text inside the footer or any other dark-background section. Use `color-border` (`#D6CFC4`, 7.5:1 on Ink) for secondary/receded text on dark backgrounds. This distinction is already documented in `SPRINT_1_FOOTER_V3_PLAN.md` §5.2.

### 4.2 Tablet Size Interpolation

Task 04 does not specify typography sizes for the 768–1023px (tablet) breakpoint. The values configured are arithmetic midpoints:

| Style | Desktop | Tablet (interpolated) | Mobile |
|-------|---------|-----------------------|--------|
| type-h1 | 52px | 44px | 36px |
| type-h2 | 38px | 32px | 28px |
| type-body | 17px | 17px (= desktop) | 16px |
| type-body-lg | 19px | 18px | 17px |
| type-quote | 24px | 22px | 20px |

These are reasonable defaults. If visual review at 768–1023px reveals that headings feel too large or too small, adjust the tablet value in Elementor Site Settings → Global Typography without changing desktop or mobile values.

### 4.3 System Colors and System Typography Left as Elementor Defaults

Elementor's 4 built-in system colors (Primary `#6EC1E4`, Secondary `#54595F`, Text `#7A7A7A`, Accent `#61CE70`) remain configured as Elementor factory defaults. They will appear in the widget color picker in the "System Colors" section — these should not be used in any template.

**In the Elementor widget color picker, our tokens appear in the "Custom Colors" section**, below the system colors. When building or editing any widget, always select from the Custom Colors section, not the System Colors section.

The same applies to system typography (Primary/Secondary/Text/Accent — Roboto/Roboto Slab): always select from the "Custom Typography" section in the widget typography dropdown.

### 4.4 Elementor CSS Not Yet Regenerated

The script deleted the cached CSS files and the `_elementor_css` meta. The new CSS reflecting the 9 colors and 5 typography styles will be generated the first time Elementor loads them. However, for the tokens to reliably appear in the widget dropdowns within the Elementor editor, a one-time manual Save is recommended (see Section 5).

## 5. Required Manual Steps

**One manual step is required before Phase C (header build) begins.** Everything else in this phase was completed programmatically.

### Step 1 — Trigger Elementor CSS Regeneration (REQUIRED)

1. Open the local site in your browser: `http://georgeungureanu-doctor-dev.local/wp-admin`
2. Navigate to: Elementor → Site Settings (in the WordPress admin sidebar)
3. This will open the Elementor panel. You do not need to change any settings.
4. Click **"Save Changes"** (the Save button in the Elementor panel)
5. Wait for the save to complete — Elementor will regenerate all CSS files with the new token definitions.

**What this does:** Elementor reads `_elementor_page_settings` from the database, generates the CSS variables for all 9 colors and 8 typography styles, and writes the new CSS files to `wp-content/uploads/elementor/css/`.

### Step 2 — Verify Tokens Appear in Widget Dropdowns (VERIFICATION)

After saving:

1. Open any page (or the header template) in the Elementor editor
2. Add a temporary Heading widget or click an existing widget
3. In the Style tab, click the color picker — verify:
   - "Custom Colors" section lists all 9 token names (color-ink, color-surface, etc.)
4. In the typography control, click the typography picker — verify:
   - "Custom Typography" section lists all 8 style names (type-h1, type-body, type-overline, type-cta, type-nav, etc.)
5. Remove the temporary widget if you added one — do not save any template changes

### Step 3 — Verify No CSS Regressions (OPTIONAL)

Load the site homepage (the "Sample Page" at `/sample-page/`) and check:

- The GU Design System plugin stylesheet loads (DevTools → Network → `gu-design-system` CSS present)
- Google Fonts load (Network → `fonts.googleapis.com` request present)
